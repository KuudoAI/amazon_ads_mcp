# Amazon Ads MCP Server — v0.3.0

## Highlights

This release transforms the `amazon_ads` MCP server from a working API
wrapper into agent-ready infrastructure. Three themes run through the
work:

- **Honest behavior** — every claim the server makes is now observable.
  The prior `persist: True` no-op is gone; session scope is signalable
  with a dedicated probe; alias rewrites fire on every request path
  including the sandbox `call_tool` hop.
- **Machine-readable schema** — `report_fields` is the only structured
  v1 reporting catalog that exists outside Amazon's HTML docs, with
  pre-flight validation that catches unknown fields, missing required
  companions, and incompatible pairs in one call.
- **Ergonomic surface** — agents can probe, validate, and iterate
  without burning round-trips. `try/except RuntimeError` around
  `call_tool` actually works in the sandbox; `set_context` collapses
  three setup calls into one; sessions persist where the transport
  supports it.

The headline: an LLM agent constructing an Amazon Ads v1 report can now
validate its query before submission and get back unknown fields,
missing required companions, and incompatible pairs — all in one call,
with accurate suggested replacements. That capability doesn't exist
anywhere else in the Amazon Ads ecosystem.

## New: `report_fields` — machine-readable v1 reporting catalog

Amazon's OpenAPI spec for the v1 reporting endpoint constrains
`query.fields` only as "array of dotted strings." The actual valid
field names, type information, and compatibility matrix live in
Amazon's HTML documentation — unreachable from any programmatic
interface. This release closes that gap.

**What it is.** A single tool with two modes:

```python
# Discovery
report_fields(mode="query", category="metric", search="cost", limit=20)

# Pre-flight validation
report_fields(
    mode="validate",
    validate_fields=["date.value", "metric.totalCost", ...],
)
```

**What it contains.** 700 metric fields plus dimensions, each with:
canonical field ID, data type, description, required companion
fields, complementary fields, full compatibility matrix (which
dimensions the metric can appear with), incompatibility matrix
(which it cannot), and cross-version mapping to v3 DSP and v3
Sponsored Ads field names.

**How it was built.** Scraped from Amazon's reporting docs as
Markdown, parsed into structured records with provenance-stamped
lineage back to the source page. The catalog carries
`catalog_schema_version`, `parsed_at`, and `stale_warning`
(configurable via `LIST_REPORT_FIELDS_STALE_DAYS`, default 90) so
consumers can detect drift.

**Validation catches three failure classes in one call:**

- **Unknown fields** — the field doesn't exist in Amazon's v1
  catalog.
- **Missing required** — e.g. `metric.totalCost` requires
  `budgetCurrency.value` as a companion.
- **Incompatible pairs** — e.g. `metric.combinedPurchases` cannot be
  combined with `audienceSegment.*` dimensions.

**Suggested replacements use fuzzy matching that actually works.**
`metric.cost` → `metric.totalCost` as the first suggestion.
`metric.purchases7d` → `metric.purchases`. `metric.sales7d` →
`metric.sales`. The alternative is what agents were doing before:
trial-and-error report submissions, each costing a real Amazon API
call.

**Compatibility queries.** `compatible_with=["Ad"]` returns the 649
fields queryable alongside the `Ad` dimension. The catalog accepts
both display labels (`"Search term"`) and field IDs
(`"searchTerm.value"`) — the server normalizes inputs so agents
don't guess which naming convention to use.

## New: `get_session_state` — honest session-scope signaling

Earlier releases silently ignored a `persist: True` parameter on
`set_context`. Context set in one execute block didn't survive to the
next, with no way for an agent to know why. This release ships honest
behavior and a dedicated probe.

`get_session_state` is a zero-cost, side-effect-free probe returning
exactly three fields:

```json
{
  "session_present": true,
  "state_scope": "session",
  "state_reason": null
}
```

`state_scope` is `"session"` when context will persist across calls,
`"request"` when it must be re-established per call. `state_reason`
enumerates why scope is constrained when applicable:

- `"no_mcp_session"` — transport has no long-lived session (e.g.
  stateless HTTP).
- `"token_swapped"` — a different bearer/refresh token arrived
  mid-session and the previous tenant's state was cleared. `state_scope`
  stays `"session"` but context must be re-established for the new
  tenant.
- `"bridge_unavailable"` — reserved; treat as `"request"`.

**Single predicate for agents:**

> Re-establish context before the next tool call iff
> `state_scope == 'request'` or `state_reason` is not null.

One probe per block is sufficient; scope cannot change within a block.

**Session state now actually persists across `execute` blocks** on
transports that support it. Set context once at the start of a
conversation; subsequent blocks inherit identity, region, and profile.

**Where the three fields appear (deliberately narrow surface).** The
fields ship on six tool responses: `set_context`,
`set_active_identity`, `get_active_identity`, `get_active_profile`,
`get_routing_state`, and the dedicated `get_session_state` probe.
Pure-data responses (`SetProfileResponse`, `SetRegionResponse`,
`ClearProfileResponse`, `ListReportFieldsResponse`, etc.) intentionally
do **not** carry the fields — pinning that contract is a parametrized
test so future tools can't quietly bloat their schemas. Tool
descriptions on every stateful surface (`set_context`,
`set_active_identity`, `set_active_profile`, `set_region`,
`clear_active_profile`, `get_session_state`, plus the `execute`
meta-tool) document the same predicate and the same enumerated
reasons.

**`token_swapped` plumbing.** A new per-request ContextVar
(`_state_reset_reason_var` in `auth/session_state.py`) is set by
`RefreshTokenMiddleware` when the bearer fingerprint changes
mid-session, and cleared in the same middleware's `finally` block so
it does not leak across requests. A new `compute_session_state(ctx)`
helper in `auth_session_bridge.py` is the single source of truth that
all six tool wrappers read.

## New: `set_context` — one-call setup

Replaces the three-call sequence (`set_active_identity` +
`set_region` + `set_active_profile`). Any subset is allowed; accepts
both `region` (preferred) and `region_code` (legacy alias). Combined
with session-state persistence, typical agent setup drops from three
calls per block to one call per session.

## New: `drop` parameter on `report_fields` — response shape control

For workflows where compatibility arrays aren't needed (autocomplete,
name-only enumeration, existence checks), callers can strip named
top-level keys from every field record:

```python
report_fields(
    mode="query",
    category="metric",
    search="cost",
    limit=20,
    drop=["compatible_dimensions", "incompatible_dimensions"],
)
```

**Measured savings:**

| Query                                      | Default        | With `drop`   | Reduction |
|--------------------------------------------|----------------|---------------|-----------|
| Search 20 records, drop both compat arrays | 23,874 chars   | 8,354 chars   | 65%       |
| Browse 50 records, drop both compat arrays | 57,906 chars   | 15,606 chars  | 73%       |
| Detail lookup, drop 4 keys                 | 1,735 chars    | 602 chars     | 65%       |

Unknown keys in `drop` raise `INVALID_MODE_ARGS` with the full
allowlist echoed inline — a deliberate typo self-documents the valid
options. `drop` passed to `mode="validate"` is rejected with a clear
explanation (validate carries no field records to shape).

## New: `download_export` + `read_download` — report retrieval round-trip

Fixed end-to-end workflow for Amazon report delivery. Previously, a
completed report could be downloaded server-side but lived outside the
active profile's directory, so `get_download_url` couldn't locate it
and the catalog's Python sandbox couldn't read it. This release fixes
the pipeline:

- `download_export` writes to
  `/app/data/profiles/{profile_id}/reports/s3-reports/...`
  (profile-scoped).
- `list_downloads` sees profile-scoped files.
- `get_download_url` returns a reachable URL.
- New `read_download` tool allows direct read of profile-scoped files
  with offset/length/encoding support — enabling in-sandbox
  processing without external network I/O.

## Enhanced: Sandbox runtime for `execute`

`try`/`except RuntimeError` now works around `call_tool` invocations.
An agent can probe multiple candidates in one block:

```python
candidates = ["metric.spend", "metric.cost", "metric.totalCost"]
for name in candidates:
    try:
        r = await call_tool("allv1_AdsApiv1CreateReport", {...})
    except RuntimeError as e:
        continue  # try next
```

Previously the first failure aborted the entire block and required N
separate round-trips to probe N candidates. This is the single biggest
ergonomic improvement for defensive/exploratory agent code.

The `execute` tool description fully documents sandbox capabilities:

- **Available scope:** `await call_tool(name, params)` with catchable
  `RuntimeError`.
- **Guardrails explicitly listed:** no network, no filesystem, no
  `asyncio.sleep`, `print` may be discarded.
- **Auth/region/profile model:** set once via `set_active_*`, rides
  every subsequent call.
- **Session-scope contract:** new section pointing agents at
  `get_session_state` plus the verbatim re-establishment rule.
- **Gotchas:** `json.dumps(default=...)` with Pydantic models, `with`
  works for pure-Python context managers only.

## Fixed: Singular-ID alias across all request paths

The v1 API's `RetrieveReport` and `DeleteReport` endpoints require
`reportIds: [...]` arrays even when cardinality is always one. Agents
naturally reach for `reportId: "..."` singular. This release accepts
both, with plural canonical winning on precedence.

**The architectural fix matters:** the alias rewrite now runs on every
request path. Initial implementation landed at the FastMCP sidecar
layer (verified by 608 unit tests) but didn't fire on the sandbox path
used by `amazon_ads:execute → call_tool`. Fixed by applying the
rewrite in `AuthBridgingSandboxProvider.bridged_call_tool` as well.

**Additional structural fix:** alias rules are now source-controlled.
Previously `openapi/resources/` was gitignored and the
`process_openapi_specs.py` regen script ran from scratch on each
deploy, silently zeroing alias rules. Now committed to the repo and
surviving clean deploys.

Aliases applied to scalar-ID endpoints (`RetrieveReport`,
`DeleteReport`). Compound-object endpoints like
`DSPRetrieveCommitmentSpend` take structured arrays where a singular
alias doesn't map cleanly — deliberately scoped out; plural canonical
remains required there.

## Fixed: Catalog compatibility graph populated

An earlier catalog release shipped with `compatible_dimensions` and
`incompatible_dimensions` `null` on every record, silently disabling
`compatible_with=[...]` queries and `incompatible_pairs` detection in
validate mode. Traced to a parser/ingestion gap — the doc parser
wasn't extracting the compatibility bullets from Amazon's HTML tables.

Fixed. All 700+ metric fields now carry full compatibility data.
`metric.combinedPurchases` alone has 40 compatible and 10 incompatible
dimensions. Compatibility validation in `mode="validate"` now actually
detects pair conflicts pre-submission.

---

## Fixed: `ClientId` auto-injection for `allv1_*` tools

Previously required explicit `Amazon-Ads-ClientId` parameters on every
call. Identity activation didn't populate the LWA Client ID header
even when `credentials_loaded: true` was returned. Fixed —
`set_active_identity` now resolves `ClientId` via the
`remote_identity_type_application` relationship and injects it at
request time. `allv1_*` tools work without any explicit auth
parameters.

---

## Fixed: Null-drop at response serialization boundary

Every `report_fields` response previously carried ~6 null-valued keys
per record (v3 names when not requested, empty compatibility arrays,
source metadata on listing paths). Now dropped at the Pydantic
serialization boundary via `exclude_none=True`.

**Impact:**

- Listing path: 26,837 chars → 23,874 chars (−11%, 121 null
  occurrences eliminated).
- Detail path: ~2,100 chars → 1,735 chars.
- Validate path: ~280 chars → 197 chars.

Empty collections (`[]`, `""`) are preserved — `[]` signals "checked,
none apply," which is information. Listing-mode responses drop
`description` and `source` (scanning context); detail lookups keep
both. `include_v3_mapping=False` now drops v3 keys entirely rather
than emitting them as `null` — stable record shape, opt-in data.

## Fixed: `set_region` parameter stability

`set_region` alternated between requiring `region` and `region_code`
across deploys. Now documented as `region` preferred with
`region_code` accepted as legacy alias — both paths validated.

## Fixed: Pre-flight validation wiring

PR-2 originally shipped the alias transform function and unit tests
validating it, but the transform registration hook looked for
`server.transform_tool` which had been removed upstream in current
FastMCP. Tests passed; the transform never actually ran. Fixed by
hooking `on_call_tool` directly via `SidecarTransformMiddleware`.

## Fixed: Download-URL path convention

`download_export` previously wrote to
`/app/data/reports/s3-reports/...` (global), but `get_download_url`
validated against `/app/data/profiles/{profile_id}/...`
(profile-scoped). The two tools had inconsistent path conventions,
producing a "successfully downloaded, cannot be located" dead-end.
Path conventions unified.

## Enhanced: Advertiser account resolution

Profile IDs (e.g. `3194577320222553`) and advertiser account IDs
(e.g. `amzn1.ads-account.g.4s485yezm24lzzsayufh0sdyh`) are distinct
identifier spaces. The v1 reporting endpoint rejects profile IDs in
`accessRequestedAccounts`. Now documented in the
`allv1_AdsApiv1CreateReport` tool description with the
`allv1_AdsApiv1QueryAdvertiserAccount` resolution path explicitly
referenced.

## Breaking changes

- **`get_active_identity` response wire shape.** Previously returned
  the raw `Identity` object. Now returns `GetActiveIdentityResponse`,
  which wraps the identity alongside the three state fields:
  ```json
  {
    "success": true,
    "identity": { "id": "...", "type": "...", "attributes": { ... } },
    "session_present": true,
    "state_scope": "session",
    "state_reason": null
  }
  ```
  Callers that read `result.id` directly must read
  `result["identity"]["id"]` instead. This is the only meaningful
  wire-shape break in the release.

- **`persist` parameter removed** from `SetActiveIdentityRequest`,
  `set_context`, and `set_active_identity`. The parameter was a no-op
  in prior releases. **Pydantic v2 silently ignores unknown fields**,
  so existing calls passing `persist=True` continue to construct
  successfully — the value is dropped, matching the prior observable
  behavior. No runtime break for typical callers; agents and tools
  reading the schema (which no longer advertises `persist`) should
  drop the parameter from their templates. Session persistence is now
  automatic where the transport supports it; check `state_scope` /
  `state_reason` on the response to confirm.

## Known follow-ups

- **`DSPRetrieveCommitmentSpend` compound-object alias.** The alias
  fires but leaves top-level artifacts that produce Amazon's cryptic
  "Expected null" error. Scalar-ID aliases are clean; nested-object
  aliases need additional scrubbing. Deferred pending real-usage
  demand.
- **`allv1_*` tool description normalization.** `CreateReport`,
  `RetrieveReport`, and `report_fields` set the quality bar. The
  remaining ~130 deferred `allv1_*` tools carry bare OpenAPI-derived
  descriptions (e.g. `DeleteReport: "Delete a report by ID"`). FastMCP
  3.2.4's docstring-to-parameter extraction could close this gap
  without manual edits — re-run the OpenAPI-to-tool generator with
  docstring-aware extraction.
- **`.build/` gitignore.** The `process_openapi_specs.py` regen script
  still lives in a gitignored directory; local merge-aware fixes can't
  ship. Either un-ignore `.build/scripts/` or move the regen into
  `src/amazon_ads_mcp/build/`.
- **SP server discovery parity.** The `amazon_sp` server surfaces only
  `execute`; the `amazon_ads` discovery pattern (tags / search /
  get_schema) hasn't been ported. Same investment that made Ads
  discoverable would help SP. (Note: the session-scope fixes apply to
  both servers automatically — both share `code_mode.EXECUTE_DESCRIPTION`,
  so this is no longer a separate follow-up.)

## Measurement discipline

This release shipped with verified measurements rather than estimates.
Where earlier proposals projected specific savings, those numbers were
measured against live endpoints before PR descriptions were finalized:

- Null-drop savings projected 20–40%; measured 11%.
- `drop` parameter savings projected 74–80%; measured 65–73%.
- Token-count measurements captured pre/post for every optimization.

The gap between estimate and measurement reflected JSON structural
overhead (envelopes, record wrappers) not captured in per-key byte
accounting. Future PRs will use the committed benchmark script rather
than per-key math.

## How to upgrade

Drop-in for most callers; remove `persist` from any `set_context` /
`set_active_identity` calls (it's silently ignored, but cleaner to
omit). If your code reads `get_active_identity` results directly, swap
`result.id` / `result.type` for `result["identity"]["id"]` /
`result["identity"]["type"]`.

```bash
pip install --upgrade amazon-ads-mcp==0.3.0
```

For agents, the recommended pattern is:

```python
state = await call_tool("get_session_state", {})
if state["state_scope"] == "request" or state["state_reason"] is not None:
    await call_tool("set_context", {"identity_id": ..., "region": ..., "profile_id": ...})
# proceed with the rest of the block; one probe per block is sufficient
```

## Acknowledgments

The `report_fields` tool exists because Amazon doesn't publish their
v1 reporting schema in machine-readable form. Every field catalog
entry traces back to a specific Amazon documentation page, parsed and
structured so agents can query it deterministically. The combination
of that catalog, pre-flight validation, session state, catchable
sandbox exceptions, and singular-ID aliases is what transforms this
MCP server from "works if you know all the edge cases" into "an LLM
can use it correctly on the first try."
